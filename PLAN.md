# תוכנית בנייה — סוכן יועץ פיננסי אישי אוטונומי + לייב דשבורד

מסמך מפרט להרצה ב-Claude Code. המטרה: מערכת מקומית שאוספת אוטומטית את כל התנועות (עו"ש + כרטיסי אשראי), בונה מודל מלא של ההתנהלות הכספית, ומריצה סוכן יועץ אוטונומי לצד דשבורד חי — לניטור, זיהוי כפילויות ומנויים, חיזוי תזרים, וייעוץ פרואקטיבי לצמצום הוצאות ולעמידה ביעדי חיסכון.

**עקרון-על:** הסוכן מייעץ ומתריע, לעולם לא מזיז כסף. כל מספר מגיע מ-SQL דטרמיניסטי; ה-LLM רק מנתח, מתעדף ומנסח.

**החלטה אדריכלית שנקבעה:** גישה לנתונים = קריאה ישירה ל-`israeli-bank-scrapers` מהקוד (Phase 1–3), לשליטה מלאה בנירמול ובדדופ. MCP נשאר כאופציה בלבד, למשיכת נתונים on-demand תוך כדי שיחה (סעיף 16).

## שלוש החלטות הליבה

1. **ניתוח על Features + מספרים דטרמיניסטיים.** הסוכן לא רואה תנועות גולמיות אלא "Financial Brief" מובנה ש-SQL מחשב. כל סכום/תחזית/חיסכון מחושב ב-SQL — ה-LLM רק מתעדף ומנסח. התוצאה: דיוק, עלות נמוכה, פרטיות, ואפס הזיות מספריות.
2. **Generator → Evaluator על ההמלצות.** סוכן אחד מנסח המלצה; סוכן שני מבקר אותה (ביסוס בנתונים, ישימות, בטיחות) לפני שהיא מוצגת. איכות במקום כמות.
3. **יעדים + זיכרון = יועץ שמלווה, לא מדווח.** מגדירים יעדי חיסכון; הסוכן עוקב, מציע צעד מתקן כשסוטים, וזוכר מה קיבלת ומה דחית כדי להשתפר עם הזמן.

# חלק א׳ — התשתית (שכבת הנתונים)

## 0. איך להשתמש במסמך

* שמור בשורש הריפו כ-`PLAN.md`. שים תמצית של "שלוש החלטות הליבה" וסעיפים 1, 1.1, 5, 17 גם ב-`CLAUDE.md` כדי שהכללים הקריטיים יישארו בהקשר לאורך הבנייה.
* בנייה בשלבים לפי חלק ג׳. בסוף כל שלב יש קריטריון קבלה — אל תתקדם לפני שהנוכחי עובד באמת.

## 1. עקרונות מנחים ואבטחה (לא לפשר)

* הכל מקומי. DB, קבצים, לוגים ומודלים — על המכונה בלבד. אין העלאת נתונים פיננסיים לענן.
* קרדנציאלס ב-Keychain של ה-OS או `.env` שב-`.gitignore`. אף פעם לא בקוד ולא ב-git.
* שכבת ה-LLM מקבלת כברירת מחדל רק אגרגציות וסיכומים. שליחת raw data מאחורי flag מפורש בלבד (סעיף 18).
* קריאה בלבד. אין פעולות תשלום/העברה בשום מקום במערכת.

## 1.1 אבטחת השליפה האוטומטית (מחמיר — קריטי)

השליפה רצה עם הקרדנציאלס האמיתיים שלך לבנק ולכרטיסים, אז שכבה זו מוקשחת בנפרד. מודל האיום: (א) גניבת קרדנציאלס דרך תלות npm שנפרצה, (ב) דליפה ללוגים/git, (ג) נתונים על דיסק לא-מוצפן. שלושתם מטופלים כאן.

**קרדנציאלס:**

* לאחסן ב-OS keychain (macOS Keychain / Windows Credential Manager / libsecret). `.env` רק כ-fallback, מחוץ ל-repo, עם `chmod 600` ו-`.gitignore`.
* לעולם לא להעביר סיסמאות כארגומנטים ל-CLI (נראים ב-`ps` ובהיסטוריית ה-shell) ולא לכתוב ללוגים.
* לטעון לזיכרון רק למשך השליפה ולנקות אחריה. טוקן ארוך-טווח (למשל OneZero) = סוד לכל דבר → keychain/מוצפן.
* אם הבנק מאפשר משתמש "צופה"/הרשאת קריאה בלבד — להשתמש בו במקום במשתמש הראשי.

**שרשרת אספקה (החשוב ביותר כאן):** תלות זדונית = גישה לקרדנציאלס הבנק. לנעול גרסאות (lockfile מחייב), להריץ `npm audit`, ולא לעדכן תלויות אוטומטית — כל bump עובר review לפני מיזוג.

**בידוד ורשת:**

* להריץ את הסקרייפר בקונטקסט מבודד (משתמש OS ייעודי או קונטיינר).
* Egress allowlist: תהליך השליפה יכול לצאת רק לדומייני הבנק/הכרטיס. כל יתר ה-outbound חסום — חוסם exfiltration גם אם תלות נפרצה.

**נתונים ולוגים:**

* קובץ ה-SQLite: `chmod 600`, על דיסק מוצפן (FileVault/BitLocker/LUKS). אופציונלי SQLCipher להצפנת ה-DB עצמו.
* לוגים בלי קרדנציאלס/OTP/מספרי חשבון מלאים — מיסוך ל-4 ספרות אחרונות בלבד.

**תפעול:** Fail closed בשגיאה · ריווח בין הרצות וכיבוד rate limits כדי לא לעורר חסימת הונאה/נעילה · Kill switch לביטול מיידי + יכולת רוטציית סיסמה · אין גיבוי לא-מוצפן, ובשום אופן לא לענן.

## 2. ארכיטקטורה

```
Data Access — israeli-bank-scrapers (ישיר מהקוד; ראה 1.1 להקשחה)
  → Normalize + Dedup → SQLite (מקור אמת יחיד)
  → Feature Engineering (SQL) → "Financial Brief" מובנה
  ├─→ Advisor Agent (Generator→Evaluator) → Recommendations + Alerts + Memory
  └─→ Live Dashboard (קורא ישירות מה-DB, בזמן אמת)
      ↑ Triggers (מתוזמן + event-driven)
```

הדשבורד והסוכן הן שתי שכבות עצמאיות מעל אותו DB — הדשבורד רץ ומתעדכן גם כשהסוכן לא פעיל. (MCP אינו בזרימת הליבה; אופציונלי בלבד — סעיף 16.)

## 3. סטאק מומלץ

* Node.js ≥ 22.12 + TypeScript.
* גישה לנתונים (נבחר): `israeli-bank-scrapers` ישירות מהקוד. אם Isracard/Amex חוסמים — הפורק המתוחזק עם דפדפן anti-detect.
* סודות: OS keychain (נבחר: `@napi-rs/keyring` — מתוחזק פעיל; החלפה ל-keytar הנטוש). `better-sqlite3` — מקור אמת מקומי, מהיר וסינכרוני.
* Dashboard: React 19 + Vite 8 + Tailwind 4 + shadcn/ui (Radix) + Recharts. ה־API נשאר Node מקומי ללא framework, והממשק הישן נשמר כ־fallback.
* אורקסטרציה/טריגרים/התראות: `node-cron`, או n8n לתזמון וניתוב התראות ל-Notion/Telegram.
* `@anthropic-ai/sdk` לסוכן. אופציונלי Ollama למודל מקומי בשלבים שנוגעים ב-raw data.

## 4. סכמת נתונים (SQLite)

* `accounts` — `id, type(checking|card), provider, display_name`
* `transactions` — `id, account_id, date, charge_date, amount, currency, amount_ils, raw_description, normalized_merchant, category, dedup_hash UNIQUE, source, is_transfer, is_fee, is_fx, installment_current, installment_total, memo`
* `subscriptions` — `id, merchant, avg_amount, cadence, first_seen, last_seen, status(active|price_increased|forgotten|trial_converted)`
* `goals` — `id, title, type(save_by_date|cut_category|cap_monthly), target_amount, category, deadline, progress, status`
* `recommendations` — `id, created_at, title, rationale, est_saving_ils, effort(low|med|high), confidence, status(new|accepted|dismissed|done), realized_saving_ils`
* `alerts` — `id, created_at, type, severity, message, related_tx_id, dismissed`
* `agent_memory` — `id, key, value, updated_at`

## 5. דדופ ומלכודות ישראליות (הלב של אמינות הנתונים)

* מפתח דדופ: `sha256(date | amount | normalized_description | account_id)`, `INSERT OR IGNORE`.
* **כפל חיוב אשראי מול עו"ש (קריטי):** הוצאות הכרטיס מופיעות גם בדף הכרטיס וגם כחיוב חודשי מרוכז בעו"ש. מקור האמת = הסקרייפר של הכרטיס; את החיוב המרוכז בעו"ש מסמנים `is_transfer=1` ומחריגים מההוצאות. בלי זה — ספירה כפולה.
* **תנועות פנימיות (לא הכנסה ולא הוצאה):** כסף שרק זז בין כיסים של המשתמש מסומן `is_transfer=1` בשני הכיוונים ומוחרג מחישובי הכנסה/הוצאה — פקדונות, חסכונות, ו**קרן הלוואה** (`הלואה קרן`), שנוחתת כזוג ±מקזז באותו יום: כסף שנכנס ומיד יוצא. כסף מושאל אינו הכנסה. הכלל מעוגן ב-`INTERNAL_TRANSFER_PATTERNS` ב-`src/ingest/merchants.ts`.
  * **החזר הלוואה הוא הוצאה אמיתית** (`הו"ק הלוואה`) — הדפוס מעוגן ב-`קרן` בכוונה, כדי שההחזר החודשי לא יימחק בטעות. ריבית על פקדון היא הכנסה אמיתית, לא transfer.
* **תשלומים:** לשמור `installment_current/total`; לתזרים לחזות תשלומים עתידיים שטרם חויבו.
* **נירמול בתי עסק:** שמות עברית מלוכלכים — שכבת נירמול (regex + מיפוי ידני) לפני קטגוריזציה/recurring.
* **מט"ח ומועדי חיוב:** מטבע מקורי + `amount_ils`; להבחין `date` (עסקה) מ-`charge_date` (חיוב).

# חלק ב׳ — הסוכן והדשבורד

## 6. פילוסופיית הסוכן

לולאה אוטונומית: sense → reason → recommend → notify → learn, בנויה סביב שלוש החלטות הליבה: ניתוח על Features, מספרים דטרמיניסטיים, ולולאת Generator→Evaluator.

## 7. שכבת ה-Features ("Financial Brief")

מה שה-SQL מחשב ומגיש לסוכן בכל ריצה:

* ממוצע וסטיית-תקן נעים לכל קטגוריה (3/6 חודשים) + z-score לחודש הנוכחי.
* מדד recurring לכל בית עסק (קצב, סכום, סטייה).
* תזרים: הכנסות מול הוצאות, קצב שריפה, יתרה נוכחית, התחייבויות תשלומים עתידיות.
* עמלות/ריבית מזוהות (`is_fee`) ועסקאות מט"ח עם מרווח המרה (`is_fx`).
* Top movers: הקטגוריות/הסוחרים שהשתנו הכי הרבה מול החודש הקודם.

## 8. יכולות אנליטיות (כל אחת מכומתת ב-₪)

* אנומליות — חריגה סטטיסטית (z-score) לפי קטגוריה/סוחר.
* כפילויות/חיוב כפול — אותו סוחר+סכום בטווח ימים קצר.
* מודיעין מנויים — חדש, התייקרות מנוי קיים, מנוי נשכח, ניסיון שהפך בתשלום.
* עמלות וריבית — עמלות בנק, ריבית משיכת יתר, דמי כרטיס. ROI גבוה, לרוב מתחת לרדאר.
* מרווח המרת מט"ח — עלות עודפת בעסקאות/משיכות בחו"ל.
* תחזית תזרים — יתרת סוף חודש + אזהרה מוקדמת למחסור.
* דליפות כסף — חיובים קטנים חוזרים שמצטברים.
* מגמות — MoM/YoY ועונתיות (חגים).
* מעקב יעדים — התקדמות מול יעדי חיסכון (סעיף 10).

## 9. מנוע ההמלצות (Generator → Evaluator)

* Generator מנסח מה-Brief: כותרת, נימוק, חיסכון משוער ב-₪, מאמץ, ביטחון.
* Evaluator (סוכן שני) מבקר לפני הצגה: המספרים נתמכים ב-SQL? ישים ולא כללי? אין הזיה/ייעוץ מסוכן? רק מה שעובר נשמר.
* תעדוף לפי חיסכון × קלות, עם הסבר שקוף לכל המלצה.

## 10. מודול יעדים וליווי

* יעדים: "לחסוך ₪X עד תאריך", "לקצץ קטגוריה ב-20%", "תקרה חודשית".
* הסוכן עוקב, מציג התקדמות, וכשסוטים מציע צעד מתקן קונקרטי. ליווי פרואקטיבי, לא רק דיווח.

## 11. זיכרון ולמידה

* `agent_memory`: יעדים, העדפות, והיסטוריית מה התקבל/נדחה.
* הסוכן לא חוזר על מה שנדחה, מכייל טון ותעדוף, ומודד חיסכון ממומש מול מוצע.

## 12. פרסונת היועץ (System Prompt מוכן)

אתה יועץ פיננסי אישי פרואקטיבי. אתה מנתח אך ורק את ה-Financial Brief המצורף — לעולם לא ממציא מספרים. תמיד קונקרטי, מכומת ב-₪, ולא שיפוטי. הקשר ישראלי (₪, חגים, עמלות בנק, חברות אשראי). לכל תובנה: מה קרה, כמה זה עולה לחודש/לשנה, ומה הצעד המדויק ומאמצו. אם אין ביסוס בנתונים — אמור שאין. קודם ההזדמנויות הגדולות והקלות.

## 13. טריגרים ואוטונומיה

* מתוזמן: משיכה יומית → תמצית; דוח שבועי; ניתוח עומק חודשי.
* Event-driven: עסקה חריגה/גדולה מעל סף → התראה מיידית; מנוי חדש/התייקרות → התראה.
* אפשר לנהל את שתי השכבות ב-n8n (cron + webhooks).

## 14. ערוצי הגשה

* תמצית יומית, התראה בזמן אמת על חיוב חריג, דוח שבועי ל-Notion/מייל/Telegram, וניתוח עומק חודשי.

## 15. לייב דשבורד — שכבה עצמאית בזמן אמת

שכבת תצוגה שרצה ומתעדכנת גם כשהסוכן לא פעיל, קוראת ישירות מ-SQLite/Features. הפרדת אחריות נקייה: הדשבורד מציג, הסוכן מנתח.

מסמך הממשק והמימוש בפועל: [`DASHBOARD.md`](DASHBOARD.md). הוא מתעד את היררכיית המידע, שפת העיצוב, התנהגות responsive והחלטות הגרפים, כדי ששינויים עתידיים ישמרו על אותה חוויית שימוש.

* עדכון חי: auto-refresh ב-polling קצר, או WebSocket שנדלק על שינוי ב-DB אחרי כל משיכה.
* מסכים: שורת KPI עליונה (יתרה, תזרים החודש, קצב שריפה, תחזית סוף חודש) · הוצאות לפי קטגוריה עם drill-down · מנויים פעילים · גרף תזרים לאורך זמן · התראות פתוחות · התקדמות יעדים · feed המלצות (accept/dismiss ישירות מהמסך, מזין את הזיכרון בסעיף 11).
* אינטראקטיבי: סינון לפי טווח/חשבון/קטגוריה, חיפוש תנועה, סימון "התעלם".
* צ'אט משובץ (אופציונלי): שאלות אד-הוק ליועץ (סעיף 12) מתוך הדשבורד.

## 16. גישה לנתונים: הישיר (נבחר) מול MCP (אופציונלי)

* אין API רשמי ליחידים — בנקאות פתוחה נעולה מאחורי רישוי רגולטורי.
* **הנבחר — קריאה ישירה מהקוד:** `israeli-bank-scrapers` נקרא ישירות ב-Phase 1–3. יתרון: שליטה מלאה בנירמול, בדדופ ובהקשחת האבטחה (1.1), והתאמה מלאה לפייפליין המובנה של הסוכן.
* **MCP — אופציה בלבד:** שרתי MCP קהילתיים (`israeli-bank-mcp` של Motti Bechhofer, `il-bank-mcp` של glekner שרץ מקומית ואף עם Ollama) חושפים לסוכן כלים כמו `fetch-transactions` ו-`two-factor-auth`. שקול רק אם תרצה שהסוכן ימשוך נתונים on-demand תוך כדי שיחה — לא כמנגנון האיסוף הקבוע.
* Caveat: קהילתי/לא רשמי, יש לטפל ב-2FA, מקומי בלבד.

## 17. אמון, guardrails והערכה

* לעולם לא מזיז כסף. קריאה בלבד. כל צעד עם השלכה — Human-in-the-loop.
* אנטי-הזיה: מספרים מ-SQL בלבד; Evaluator חוסם המלצות לא-מבוססות.
* מדד איכות: יחס המלצות שהתקבלו + סך החיסכון הממומש לאורך זמן.

## 18. שכבות פרטיות

* Tier 1 (מרבי): שלבים שנוגעים ב-raw data רצים במודל מקומי (Ollama); הענן מקבל רק Brief מאוגרג.
* Tier 2 (ברירת מחדל): ניתוח בענן על Brief מאוגרג בלבד. מעבר ל-raw בענן — רק מאחורי flag מודע.

# חלק ג׳ — תוכנית בנייה בשלבים

**Phase 0 — Scaffolding + אבטחה.** Node/TS, `.gitignore`, ניהול סודות ב-keychain, הרשאות קבצים, ושלד DB (סעיף 4). קבלה: סודות ב-keychain (לא ב-git/קוד), הפרויקט עולה.

**Phase 1 — Ingestion חשבון אחד (ישיר).** קריאה ישירה ל-`israeli-bank-scrapers` לעו"ש → SQLite עם דדופ, תחת הקשחת 1.1 (keychain + egress allowlist + לוגים ממוסכים). קבלה: הרצה חוזרת לא מכפילה; אין קרדנציאלס/OTP בלוגים.

**Phase 2 — כל החשבונות + נירמול + דדופ מלא.** כרטיסים + טיפול בכפל חיוב עו"ש (סעיף 5). קבלה: סך הוצאות תואם ידנית.

**Phase 3 — Feature layer.** בניית ה-Financial Brief ב-SQL (סעיף 7). קבלה: Brief תקין לחודש נבחר.

**Phase 4 — Advisor core.** Generator→Evaluator, טבלת recommendations, אנטי-הזיה. קבלה: מזהה ≥1 כפילות/מנוי אמיתי, מספרים תואמים SQL.

**Phase 5 — יעדים + זיכרון.** מודול יעדים וליווי, `agent_memory`, למידה מדחיות. קבלה: עוקב אחרי יעד ומציע צעד מתקן. **מומש:** תקרת הוצאות חודשית, תקרה לקטגוריה וחיסכון עד תאריך; ההתקדמות והצעד המתקן מחושבים דטרמיניסטית ונכללים ב־Financial Brief. בדיקת קבלה: `npm run goals:selftest`.

**Phase 6 — לייב דשבורד.** שכבת תצוגה עצמאית עם auto-refresh (סעיף 15). קבלה: מתעדכן בזמן אמת מה-DB, accept/dismiss עובד.

**Phase 7 — טריגרים + הגשה.** מתוזמן + event-driven, דוח שבועי/חודשי (סעיפים 13–14). קבלה: רץ לבד, מייצר דוח + התראה בזמן אמת. **מומש:** scheduler מקומי עם opt-in מפורש, lock ו־kill switch; סריקת התראות דטרמיניסטית אחרי ingest; דוחות Markdown מקומיים בהרשאות `600`. בדיקת קבלה: `npm run automation:selftest`. הפעלה ותפעול: [`AUTOMATION.md`](AUTOMATION.md).

**מצב נוכחי (2026-07-19):** Phases 0–7 ממומשים בקוד. Phase 6 עבר redesign מלא ל־React + shadcn/ui עם fallback לממשק המקורי ובדיקת דפדפן אוטומטית. דוח שבועי אמיתי נוצר מקומית בהצלחה.

**עבודות סגירה שאינן phase קוד חדש:** הפעלת ה־scheduler ובדיקת ingest מתוזמן מול החשבונות האמיתיים, קרדנציאלס ו־2FA נשארות בדיקות ידניות של המשתמש. יש להשלים/לאמת ברמת מערכת ההפעלה את בידוד תהליך השליפה, egress allowlist והצפנת הדיסק; לאמת ידנית שסך ההוצאות תואם לדוחות הבנק; ולבדוק את ה־Generator→Evaluator על המלצה אמיתית. צ׳אט משובץ, MCP, n8n ו־Ollama הם הרחבות אופציונליות ואינם חלק מקריטריון הסיום.
